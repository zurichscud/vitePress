# 非对称加密（RSA）

## 为什么需要非对称加密

对称加密中使用的密钥必须见面协商，因为密钥无法在网络传输，密钥的传输需要加密，而没有密钥则无法加密。

## 秘钥对

公钥可以公开给任何人。发送信息的人使用公钥进行加密，只有持有私钥的人才能解密。因此私钥不能在互联网中传播。

>  公钥私钥的概念是相对的：公钥私钥只是“是否公开”上的区别，在数学上，用其中一把加密，就能用另一把解密

```js
发送方：使用 公钥 加密数据 → 发送密文
接收方：使用 私钥 解密密文 → 得到原文
```

一般来说服务器会持有私钥。前端拿到公钥可以对信息进行加密

## 数字签名

但是在传输公钥的过程，存在第三方拦截公钥的情况。第三方将自己的公钥分享给你。发送方使用黑客的公钥进行加密，黑客就可以根据自己的密钥对解密你发送的信息，然后篡改你发送的信息发送至接收方。

为了保证公钥不被伪造，我们需要将公钥委托给第三方机构CA进行登记。登记公钥与身份之间的关系。

## 数字证书

数字证书（Digital Certificate）是由认证机构（CA，Certificate Authority）颁发的一种电子文档，用于验证公钥的归属以及确保证书持有者的身份。它通常包含持有者的公钥、持有者的身份信息、证书的有效期、证书颁发机构的数字签名等内容。



![image-20260302174114357](https://markdown-lai.oss-cn-hangzhou.aliyuncs.com/typora/image-20260302174114357.png)

根证书记录了可以信赖的CA机构信息和公钥。根证书预先安装在系统中可以杜绝CA机构公钥被伪造的可能。通过CA的公钥可以验证数字证书的数字签名，确保了数字证书的不可伪造性。

1. 签名：CA对登记的公钥和持有者信息使用自己的私钥进行签名制作成数字证书。

2. 客户端请求服务端时，服务端将CA颁发的数字证书发送给客户端。

3. 验签：客户端使用CA的公钥进行解密即可验证公钥的真实性（验签）。

## 签名与验签



### 第一阶段：发送方（签名）

1. **取指纹 (Hashing)**：发送方使用 Hash 算法（如 SHA-256）将原始消息生成一段固定长度的**摘要 (Digest)**。
2. **私钥加密 (Signing)**：发送方使用自己的**私钥**对这段“摘要”进行加密。这个加密后的结果就是**数字签名**。
3. **发送数字证书**：将**原始消息**和**数字签名**一起发给接收方。

### 第二阶段：接收方（验签）

1. **解密签名**：接收方使用发送方的**公钥**对收到的“数字签名”进行解密，得到**摘要 A**。
2. **重算摘要**：接收方对收到的“原始消息”用同样的 Hash 算法重新算一遍，得到**摘要 B**。
3. **对比**：如果 **摘要 A == 摘要 B**，说明：
   - 消息确实是持有私钥的那个人发的（身份有效）。
   - 消息在传输过程中没有被改动过（内容完整）。



验签一致，只能说明签名者一定是私钥拥有者，但是不能保证私钥拥有者的身份。
