

# Misfire

**Misfire（错失触发）** 描述的是任务触发器（Trigger）错过原定执行时间时，Quartz该怎么处理。简单来说，就是“任务本应该在什么时候执行，但因为某些原因没执行，现在怎么办”。

## 为什么会发生 Misfire？

通常由以下几种情况引起：

- **服务停机**：服务器重启或崩溃时，正处于触发点的任务没跑。
- **线程饥饿**：Quartz 线程池太小（比如只有 5 个线程），但同一时间有 10 个任务要跑，剩下的 5 个只能排队，等排到它们时已经过了预定时间。
- **任务阻塞**：上一个相同任务还没跑完（使用了 `@DisallowConcurrentExecution`），导致下一个任务无法准时开始。
- **系统时钟跳变**：服务器时间被手动修改。

## MisfireThreshold

Quartz 并不是一旦晚了 1 秒就判定为 Misfire。它有一个全局配置参数： `org.quartz.jobStore.misfireThreshold`（默认是 **60,000 毫秒**，即 1 分钟）。

- 如果任务晚了 **30 秒**：Quartz 认为这只是普通的延迟，会立即执行。

- 如果任务晚了 **70 秒**：Quartz 认为发生了 **Misfire**，此时会根据你配置的 **策略（Instruction）** 来决定怎么办。

## MISFIRE_INSTRUCTION

`MISFIRE_INSTRUCTION_SMART_POLICY`默认策略

- **SimpleTrigger** ：**立即触发一次**，然后按原来的重复次数和间隔继续执行。

- CronTrigger：DO_NOTHING

1️⃣ SimpleTrigger Misfire策略表

| Misfire策略                       | 行为说明                                                     |
| --------------------------------- | ------------------------------------------------------------ |
| `FIRE_NOW`                        | 立即触发一次，然后按原来的间隔继续执行                       |
| `NOW_WITH_EXISTING_REPEAT_COUNT`  | 立即执行一次，剩余重复次数不变                               |
| `NOW_WITH_REMAINING_REPEAT_COUNT` | 立即执行一次，剩余重复次数调整，**不算错过次数**【错过就算作废了】 |
| `NEXT_WITH_REMAINING_COUNT`       | 不立即执行，等待下一个时间点，剩余次数不变                   |
| `NEXT_WITH_EXISTING_COUNT`        | 不立即执行，等待下一个时间点，保持原总次数                   |

| 策略          | 结果                                                         |
| ------------- | ------------------------------------------------------------ |
| **EXISTING**  | 立即执行一次，然后继续按原计划执行，**总共还是执行 11 次**。结束时间会比原定晚。 |
| **REMAINING** | 立即执行一次，但只执行**剩余次数**（11 - 2 = 9 次），错过的 2 次作废。结束时间基本不变。 |

2️⃣ CronTrigger Misfire策略表

| Misfire策略     | 行为说明                               |
| --------------- | -------------------------------------- |
| `DO_NOTHING`    | 忽略错过的触发，等待下一次cron时间触发 |
| `FIRE_ONCE_NOW` | 立即触发一次，然后按cron表达式继续执行 |

## Example

任务开始：10:00

间隔：1小时

重复次数：3（总共执行 4 次：10:00、11:00、12:00、13:00）

服务器停机：09:50 → 11:20 重启

### FIRE_NOW

错过的任务立刻触发一次，剩余次数按原计划继续。

| 原计划时间 | 实际触发时间 | 说明                                  |
| ---------- | ------------ | ------------------------------------- |
| 10:00      | 11:20        | 服务器重启后立即触发错过的 10:00 任务 |
| 11:00      | 11:20        | 同时立即触发错过的 11:00 任务         |
| 12:00      | 12:00        | 按原计划执行                          |
| 13:00      | 13:00        | 按原计划执行                          |

### NOW_WITH_EXISTING_REPEAT_COUNT

立即执行错过任务，但间隔是以**触发时间为基准**继续计算。

实际执行 **4 次**（11:20、12:20、13:20、14:20），总次数和原计划一致，但**整体时间线后移了 1 小时 20 分钟**。

| 执行序号 | 执行时间  | 说明                         |
| -------- | --------- | ---------------------------- |
| 1        | ~~10:00~~ | ❌ 错过（宕机）               |
| 2        | ~~11:00~~ | ❌ 错过（宕机）               |
| 3        | **11:20** | ✅ 立即补偿（恢复后马上执行） |
| 4        | 12:20     | 按新间隔继续                 |
| 5        | 13:20     | 按新间隔继续                 |
| 6        | 14:20     | 按新间隔继续                 |

| 原计划时间 | 实际触发时间 | 说明                          |
| ---------- | ------------ | ----------------------------- |
| 10:00      | 11:20        | 立即执行一次错过的 10:00 任务 |
| 11:00      | 11:20        | 立即执行一次错过的 11:00 任务 |
| 12:00      | 12:20        | 下一次间隔1小时执行           |
| 13:00      | 13:20        | 下一次间隔1小时执行           |

### NOW_WITH_REMAINING_REPEAT_COUNT

**结果**：实际执行 **3 次**（11:20、12:20、13:20），比原计划**少执行 1 次**，但**结束时间基本保持原定的 13:00 附近**。

| 执行序号 | 执行时间  | 说明                               |
| -------- | --------- | ---------------------------------- |
| 1        | ~~10:00~~ | ❌ 跳过，不算了                     |
| 2        | ~~11:00~~ | ❌ 跳过，不算了                     |
| 3        | **11:20** | ✅ 立即执行（剩余 2 次中的第 1 次） |
| 4        | 12:20     | 剩余 1 次                          |
| 5        | 13:20     | 剩余 0 次，结束                    |

### NEXT_WITH_EXISTING_COUNT

错过任务直接跳过，Quartz 会自动把剩余次数补到后面执行。

| 原计划时间   | 实际触发时间 | 说明                   |
| ------------ | ------------ | ---------------------- |
| 10:00        | 跳过         | 错过，不立即执行       |
| 11:00        | 跳过         | 错过，不立即执行       |
| 12:00        | 12:00        | 第一次执行（原计划）   |
| 13:00        | 13:00        | 第二次执行（原计划）   |
| （剩余次数） | 14:00        | 第三次执行，补足总次数 |

### SMART_POLICY

对 SimpleTrigger，SMART_POLICY **等同于 `MISFIRE_INSTRUCTION_FIRE_NOW`**

所以行为同第 1 种策略：

- 11:20 重启后，立即执行 10:00 和 11:00 两次错过任务
- 后续按原计划继续执行 12:00、13:00
