# 持久化

Quartz 的 **持久化（JobStore）** 是 Quartz 的一个核心功能，它允许你把 Job、Trigger、Calendar 等信息保存到数据库里，使任务在 Scheduler 重启或集群环境下仍然能够继续执行。

Quartz 提供两种主要 JobStore：

| 类型          | 描述                                  | 使用场景                                    |
| ------------- | ------------------------------------- | ------------------------------------------- |
| `RAMJobStore` | 任务信息存储在内存中，Quartz 默认使用 | 开发测试、简单单节点，重启任务丢失          |
| `JobStoreTX`  | 基于数据库，使用 JDBC 事务            | 单节点或集群，可靠持久化                    |
| `JobStoreCMT` | 基于容器管理事务（JTA）               | 容器中管理事务，例如在 Spring 或 Java EE 中 |

## 数据库表结构

Quartz 自带 SQL 脚本（支持 MySQL、PostgreSQL、Oracle 等），主要表：

- **QRTZ_JOB_DETAILS**：存储 Job 信息
- **QRTZ_TRIGGERS**：存储 Trigger 信息
- **QRTZ_SIMPLE_TRIGGERS / QRTZ_CRON_TRIGGERS / QRTZ_SIMPROP_TRIGGERS**：存储不同类型触发器的参数
- **QRTZ_CALENDARS**：存储 Calendar 信息
- **QRTZ_FIRED_TRIGGERS**：存储已经触发的 Trigger（运行中状态）
- **QRTZ_PAUSED_TRIGGER_GRPS**：存储被暂停的 Trigger 分组
- **QRTZ_LOCKS**：集群锁表，保证同一任务只被一个节点执行

### QRTZ_JOB_DETAILS

| 字段                | 类型       | 说明                                                         |
| ------------------- | ---------- | ------------------------------------------------------------ |
| `SCHED_NAME`        | VARCHAR    | 调度器名字，Quartz 支持多调度器共享数据库                    |
| `JOB_NAME`          | VARCHAR    | Job 名称（唯一标识）                                         |
| `JOB_GROUP`         | VARCHAR    | Job 所在分组（和 JOB_NAME 组合唯一）                         |
| `DESCRIPTION`       | VARCHAR    | 描述信息                                                     |
| `JOB_CLASS_NAME`    | VARCHAR    | Job 的实现类全路径（必须实现 `Job` 接口）                    |
| `IS_DURABLE`        | VARCHAR(1) | 是否持久化，`Y` 表示持久化（即没有触发器也不会被删除）       |
| `IS_NONCONCURRENT`  | VARCHAR(1) | 是否禁止并发执行（`Y` 表示同一个 Job 在同一时间只能执行一次） |
| `IS_UPDATE_DATA`    | VARCHAR(1) | 是否允许更新 JobDataMap 数据                                 |
| `REQUESTS_RECOVERY` | VARCHAR(1) | 是否请求恢复（当 Scheduler 崩溃后，是否重新执行该 Job）      |
| `JOB_DATA`          | BLOB       | 存储 JobDataMap 的序列化对象（自定义参数等）                 |

### QRTZ_TRIGGERS

**触发器总表**，用来存储 **所有类型触发器的通用信息**，而像 `QRTZ_SIMPLE_TRIGGERS`、`QRTZ_CRON_TRIGGERS` 则是 **子表**，存储特定类型触发器的详细信息。

| 字段                      | 说明                                            |
| ------------------------- | ----------------------------------------------- |
| `SCHED_NAME`              | 调度器名                                        |
| `TRIGGER_NAME`            | 触发器名                                        |
| `TRIGGER_GROUP`           | 触发器分组                                      |
| `JOB_NAME`                | 关联 Job 名称                                   |
| `JOB_GROUP`               | 关联 Job 分组                                   |
| `DESCRIPTION`             | 描述                                            |
| `NEXT_FIRE_TIME`          | 下次触发时间（时间戳）                          |
| `PREV_FIRE_TIME`          | 上次触发时间，-1表示不触发                      |
| `TRIGGER_TYPE`            | 类型，如 `S` = SimpleTrigger, `C` = CronTrigger |
| `MISFIRE_INSTR`           | Misfire 策略                                    |
| `START_TIME` / `END_TIME` | 触发器有效时间                                  |

##  **不推荐直接操作 Quartz 数据库表来更改 Job**

### **数据一致性**

- Quartz 有多个表联动：`QRTZ_TRIGGERS`、`QRTZ_JOB_DETAILS`、`QRTZ_FIRED_TRIGGERS` 等
- 直接改表可能导致 `NEXT_FIRE_TIME` 和触发器状态不一致

### **缓存问题**

- Quartz 在内存里有缓存（RAMJobStore 缓存 Job/Trigger），直接改表不会马上生效，除非 Scheduler 重启或者触发刷新

### 使用 Quartz API 修改 Job/Trigger

```java
Scheduler scheduler = StdSchedulerFactory.getDefaultScheduler();

// 1. 修改 JobDetail
JobKey jobKey = JobKey.jobKey("myJob", "myGroup");
JobDetail jobDetail = scheduler.getJobDetail(jobKey);

JobBuilder jb = jobDetail.getJobBuilder();
JobDetail newJob = jb.withDescription("新的描述").build();

// 更新 Job
scheduler.addJob(newJob, true); // true 表示覆盖已有 Job

// 2. 修改 Trigger
TriggerKey triggerKey = TriggerKey.triggerKey("myTrigger", "myGroup");
Trigger oldTrigger = scheduler.getTrigger(triggerKey);

TriggerBuilder tb = oldTrigger.getTriggerBuilder();
Trigger newTrigger = tb.withSchedule(CronScheduleBuilder.cronSchedule("0 0/5 * * * ?")).build();

scheduler.rescheduleJob(triggerKey, newTrigger);
```
